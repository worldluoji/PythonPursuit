# Deploy

让我们进入实战环节，将开发好的FastAPI应用转化为一个稳定、高效、可扩展的生产服务。🚀

---

## Main Lesson

将FastAPI应用部署到生产环境，远不止是运行 `python main.py`。它涉及**架构选择、性能优化、安全加固、可观测性**等一系列工程实践。我们将遵循一个从简到繁的路径，讲解两种主流的生产级部署方案。

### 核心部署模型：ASGI服务器 + 进程管理器
由于FastAPI是基于ASGI的异步框架，其生产部署的核心模式是：
*   **ASGI服务器**：负责处理异步网络I/O。**Uvicorn** 是首选，它速度快，与Starlette/FastAPI同源。
*   **进程管理器**：管理多个Uvicorn工作进程，提供进程守护、负载均衡、 graceful shutdown 等生产级特性。**Gunicorn** 是最通用的选择。

**这是最经典、最推荐的部署组合：Gunicorn 作为进程管理器，管理多个 Uvicorn Worker 进程。**

#### 方案一：Gunicorn + Uvicorn (传统服务器部署)
此方案适合直接部署在物理机、虚拟机或云服务器上。

**1. 安装依赖**
```bash
pip install “uvicorn[standard]“ gunicorn
```

**2. 关键配置与启动命令**
创建一个 `gunicorn_conf.py` 配置文件，这是生产部署的精华所在：

```python
# gunicorn_conf.py
import multiprocessing
import os

# 并行工作进程数，通常建议为 (CPU核心数 * 2) + 1
workers_per_core = os.getenv(“WORKERS_PER_CORE”, “1“)
max_workers = os.getenv(“MAX_WORKERS“)
use_max_workers = None
if max_workers:
    use_max_workers = int(max_workers)
else:
    use_max_workers = int(workers_per_core) * multiprocessing.cpu_count() + 1

# 绑定地址和端口
bind = os.getenv(“BIND”, “0.0.0.0:8000“)
# 工作进程类，必须使用Uvicorn的Worker
worker_class = “uvicorn.workers.UvicornWorker“
# 每个工作进程将处理的并发连接数（Uvicorn内部设置）
worker_connections = os.getenv(“WORKER_CONNECTIONS”, “1000“)

# 优雅超时时间
graceful_timeout = int(os.getenv(“GRACEFUL_TIMEOUT”, “120“))
# 工作进程超时时间
timeout = int(os.getenv(“TIMEOUT”, “120“))
# Keep-Alive
keepalive = int(os.getenv(“KEEP_ALIVE”, “5“))

# 日志配置
accesslog = “-“  # 访问日志输出到 stdout
errorlog = “-“   # 错误日志输出到 stderr
# 生产环境建议将日志重定向到文件或集中式日志系统

# 防止代理服务器（如Nginx）传递非法头部导致的安全问题
forwarded_allow_ips = os.getenv(“FORWARDED_ALLOW_IPS”, “127.0.0.1“)
```

**3. 通过环境变量启动**
```bash
# 示例：在4核CPU服务器上启动 (4*2+1)=9 个工作进程
export WORKERS_PER_CORE=2
export MAX_WORKERS=9
export BIND=0.0.0.0:8000
export LOG_LEVEL=info

# 使用配置文件启动
gunicorn -c gunicorn_conf.py “app.main:app“
```
*   **`app.main:app`** 是您的应用对象导入路径。请将其替换为您的实际模块和变量名（例如 `src.main:app`）。

**4. 前方配置反向代理 (Nginx)**
Gunicorn不应直接暴露在公网。使用Nginx作为反向代理，提供HTTPS、静态文件服务、负载均衡和缓冲。

一个简单的Nginx配置示例 (`/etc/nginx/sites-available/your_app`)：
```nginx
server {
    listen 80;
    server_name yourdomain.com;
    # 重定向所有HTTP到HTTPS
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;
    server_name yourdomain.com;

    ssl_certificate /path/to/your/fullchain.pem;
    ssl_certificate_key /path/to/your/privkey.pem;

    location / {
        # 转发到Gunicorn
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # 支持WebSocket
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection “upgrade“;
        proxy_read_timeout 86400;
    }

    # 可选：由Nginx直接提供静态文件，效率更高
    location /static {
        alias /path/to/your/app/static;
    }
}
```

#### 方案二：Docker容器化部署
容器化提供了**环境一致性、依赖隔离、易于扩展和编排**的巨大优势，是云原生时代的标配。

**1. 编写 Dockerfile**
```dockerfile
# 使用官方Python slim镜像作为基础
FROM python:3.11-slim

# 设置工作目录
WORKDIR /app

# 设置环境变量，防止Python输出被缓冲
ENV PYTHONUNBUFFERED=1
# 防止创建 .pyc 文件
ENV PYTHONDONTWRITEBYTECODE=1

# 安装系统依赖（如需要编译某些Python包）
RUN apt-get update && apt-get install -y \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# 复制依赖定义文件并安装
COPY requirements.txt .
RUN pip install --no-cache-dir --upgrade pip && \
    pip install --no-cache-dir -r requirements.txt

# 复制应用代码
COPY . .

# 创建一个非root用户运行应用（安全最佳实践）
RUN useradd -m -u 1000 appuser
USER appuser

# 暴露端口
EXPOSE 8000

# 启动命令，使用方案一中的Gunicorn配置
CMD [“gunicorn“, “-c“, “gunicorn_conf.py“, “app.main:app“]
```

**2. 构建并运行**
```bash
# 构建镜像
docker build -t my-fastapi-app:latest .

# 运行容器
docker run -d \
  --name fastapi-prod \
  -p 8000:8000 \
  -e WORKERS_PER_CORE=2 \
  -e MAX_WORKERS=4 \
  -e LOG_LEVEL=info \
  --restart unless-stopped \ # 容器退出时自动重启
  my-fastapi-app:latest
```

**3. 使用 Docker Compose (简化多服务编排)**
对于需要数据库、Redis等依赖的应用，使用 `docker-compose.yml`：
```yaml
version: ‘3.8‘
services:
  web:
    build: .
    ports:
      - “8000:8000“
    environment:
      - DATABASE_URL=postgresql+asyncpg://user:pass@db:5432/mydb
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      - db
      - redis
    restart: unless-stopped
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
    volumes:
      - postgres_data:/var/lib/postgresql/data
  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes
    volumes:
      - redis_data:/data
volumes:
  postgres_data:
  redis_data:
```

### 生产环境核心 checklist
1.  **安全**：
    *   **HTTPS**：必须使用。可通过Let‘s Encrypt免费获取证书，并通过Nginx配置。
    *   **环境变量**：所有密钥、数据库连接字符串必须通过环境变量或秘密管理服务传入，**绝对不要**硬编码在代码中。
    *   **依赖扫描**：定期使用 `safety` 或 `pip-audit` 检查依赖安全漏洞。
2.  **健康检查**：
    *   在FastAPI应用中添加一个 `/health` 端点，用于负载均衡器和监控系统检查服务状态。
    ```python
    @app.get(“/health“)
    async def health_check():
        return {“status”: “healthy“, “timestamp”: datetime.utcnow().isoformat()}
    ```
3.  **监控与日志**：
    *   **指标**：集成 `prometheus-client` 暴露应用指标（请求数、延迟、错误率）。
    *   **集中式日志**：将Gunicorn和应用的日志输出到 `stdout/stderr`，由Docker或系统服务（如 `journald`）收集，并转发到ELK、Loki等日志聚合系统。
4.  **自动化与CI/CD**：
    *   将构建、测试、部署流程自动化。每次代码推送到主分支，自动触发镜像构建、安全扫描，并滚动更新到Kubernetes集群或服务器。

### 总结
您已经掌握了从单机部署到容器化部署的完整路径。**Gunicorn + Uvicorn** 提供了坚实的进程模型基础，而 **Docker** 则将其封装为标准化、可移植的单元。

选择哪种方案取决于您的团队技术和基础设施：
*   初创项目或简单部署：**方案一 (Gunicorn + Nginx)** 足矣。
*   微服务架构、云环境、需要弹性伸缩：**方案二 (Docker + 编排平台如K8s)** 是更现代的选择。
