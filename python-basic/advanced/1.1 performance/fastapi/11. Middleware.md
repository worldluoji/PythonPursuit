# Middleware

### 中间件 vs. 依赖注入：定位与区别

| 特性 | **中间件** | **依赖注入** |
| :--- | :--- | :--- |
| **执行阶段** | 在**整个ASGI应用的最外层**运作，是请求进入、响应离开的**第一道和最后一道关卡**。 | 在**特定的路径操作函数内部**执行，是业务逻辑的一部分。 |
| **处理时机** | 在**路由匹配发生之前**就已经执行。它可以访问请求，但此时**还不知道**是哪个路径操作会处理它。 | 在**路由匹配完成之后**执行，明确知道是哪个端点将被调用。 |
| **主要目的** | 处理**跨所有请求的、与业务逻辑无关**的横切关注点，如CORS、全局日志、添加安全头、计算耗时、修改请求路径/头。 | 处理**与特定业务端点相关**的通用预处理，如认证、授权、数据库会话获取、权限检查。 |
| **访问范围** | 操作原始的请求和响应。可以**阻止请求**继续传递（如返回一个403错误），或**修改请求/响应**的原始信息。 | 操作的是被解析后的请求参数（如Pydantic模型、查询参数），并直接影响注入到路径操作函数中的值。 |
| **性能影响** | 应保持**极其轻量**，因为每个请求都会经过所有中间件。 | 可以根据需要进行复杂的逻辑和I/O操作（如数据库查询），但需注意性能。 |

### 一个简单的中间件示例：计算请求耗时并添加头部
```python
import time
from fastapi import FastAPI, Request

app = FastAPI()

@app.middleware(“http”)
async def add_process_time_header(request: Request, call_next):
    “”“
    自定义中间件：
    1. 记录请求开始时间。
    2. 将请求传递给`call_next`（即下一个中间件或最终应用）。
    3. 计算耗时，并添加到响应头中。
    ”“”
    # 1. 请求到达时
    start_time = time.time()

    # 2. 将请求传递下去，并等待响应
    response = await call_next(request)

    # 3. 得到响应后，进行处理
    process_time = time.time() - start_time
    response.headers[“X-Process-Time”] = str(process_time)  # 添加自定义响应头
    print(f“请求 {request.method} {request.url.path} 处理耗时：{process_time:.3f}秒”)

    return response

@app.get(“/“)
async def read_root():
    return {“message”: “Hello World”}
```

### 关键内置中间件：CORS
**CORS**是中间件最典型的应用之一。它必须在**路由之前、响应返回前**处理，以决定是否允许跨域请求以及添加哪些CORS头。FastAPI通过 `CORSMiddleware` 使其变得非常简单：

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[“https://example.com“],  # 允许的源列表，可使用 [“*“] 允许所有（不安全，仅用于开发）
    allow_credentials=True,  # 是否允许发送Cookie
    allow_methods=[“*“],     # 允许的HTTP方法
    allow_headers=[“*“],     # 允许的请求头
)
```

### 总结与最佳实践建议
基于您的理解和上述分析，可以得出清晰的决策路径：

*   **使用中间件，当您需要**：
    *   在业务逻辑**完全不知情**的情况下处理请求/响应。
    *   操作原始的HTTP请求/响应对象（如URL、头部）。
    *   实现**全局性、基础设施级别**的功能（CORS、GZip压缩、全局速率限制、请求ID注入）。

*   **使用依赖注入，当您需要**：
    *   执行**与特定端点业务逻辑强相关**的预处理。
    *   获取**已被FastAPI解析后的参数**（如JSON体、查询参数）。
    *   实现可复用、可组合、可测试的业务逻辑单元（如认证、获取数据库会话）。

**所以，对于您提到的“登录校验”，虽然它“全局”需要，但因为其通常涉及复杂的业务逻辑（查询数据库、验证令牌、检查权限），并且需要将结果（用户对象）清晰地注入到业务函数中，所以使用**依赖注入**是更标准、更灵活的选择。** 中间件更适合做最前置的、轻量的检查和标记。
